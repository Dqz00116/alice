# Multi-Turn CLI and Tool Support for Anthropic Provider

## Problem

1. The Alice CLI currently exits after a single assistant response because stdin is read only once before the event loop. This makes it impossible to have a continuous conversation.
2. Registered tools (such as `echo`) are not sent to the Anthropic API, so the model cannot invoke them. Even though the tool execution pipeline exists end-to-end, the LLM never sees the tool definitions.

## Goals

- Allow the CLI to run multiple conversation turns until the user explicitly exits or `max_steps` is reached.
- Send registered tool definitions to Anthropic in the chat request body.
- Ensure tool results are correctly appended to the message history and trigger the next LLM call.
- Keep all existing tests passing and add new tests for multi-turn behavior and tool propagation.

## Non-Goals

- Add a DeepSeek provider or provider-agnostic tool schema.
- Implement automatic tool retry or error recovery beyond the existing `ToolSystem` behavior.
- Add streaming tool result rendering.
- Add persistent conversation history across CLI sessions.

## Acceptance Criteria

- [ ] `StreamingProvider::format_messages` signature accepts tool definitions so the request body can include a `tools` array.
- [ ] `AnthropicProvider` serializes tool definitions into the Anthropic `tools` format.
- [ ] `EffectExecutor::CallLLM` passes `ToolsComponent.definitions` to `format_messages`.
- [ ] CLI wraps the event loop in an outer loop that reads the next user input after each assistant response.
- [ ] CLI exits on commands `/exit` or `/quit` or when `LoopComponent.should_continue` becomes false.
- [ ] New integration tests verify tool definitions are included in the formatted request body.
- [ ] New CLI integration test verifies two consecutive user inputs produce two assistant responses.
- [ ] `cargo test --all-targets`, `cargo clippy --all-targets`, and `cargo build --release` all pass.
